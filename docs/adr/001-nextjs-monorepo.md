# ADR-001: Переход с Vite+NestJS на Next.js App Router Monorepo

**Статус**: Принято
**Дата**: 2025-01-15
**Автор**: CTO Agent

## Контекст

Портал Автомоек изначально проектировался как SPA (Vite+React) + REST API (NestJS). Это стандартная архитектура, но она создаёт фундаментальную проблему для SEO-ориентированного портала: поисковые роботы Яндекса плохо индексируют SPA, требуя SSR или SSG.

### Проблемы текущей архитектуры
1. **SEO**: Vite SPA = Client-Side Rendering. Яндекс-бот видит пустой HTML без контента
2. **ISR невозможен**: NestJS не умеет Incremental Static Regeneration
3. **Программатический SEO**: Генерация тысяч страниц (город × тип) требует SSG/ISR
4. **Дублирование**: два отдельных приложения усложняют деплой и типизацию

## Рассмотренные варианты

### Вариант 1: Добавить SSR к Vite через vite-plugin-ssr
- **Плюсы**: минимальные изменения
- **Минусы**: сложная настройка, нет экосистемы ISR, слабее чем Next.js для SEO

### Вариант 2: Next.js Pages Router
- **Плюсы**: зрелый, хорошо известный
- **Минусы**: App Router — будущее Next.js, Pages Router получает меньше инноваций

### Вариант 3: Next.js App Router (выбрано)
- **Плюсы**: Server Components, ISR из коробки, generateStaticParams, Server Actions, отличная Яндекс-индексация
- **Минусы**: более крутая кривая обучения

### Вариант 4: Remix
- **Плюсы**: хорошее SSR
- **Минусы**: меньше экосистема в РФ, нет ISR

## Решение

Перейти на **Next.js 14 App Router** в монорепо с npm workspaces.

## Последствия

### Положительные
- SEO-оптимизация из коробки (ISR, SSG, Server Components)
- Программатический SEO: `generateStaticParams` для матрицы город×тип
- Единый стек: нет отдельного NestJS сервера
- Server Actions заменяют REST для форм и мутаций
- TypeScript end-to-end без дублирования типов

### Отрицательные
- Нужен рефакторинг существующего кода
- Некоторые NestJS паттерны (декораторы, DI) не переносятся напрямую
- Команда должна освоить App Router паттерны

### Нейтральные
- PostgreSQL остаётся, меняется только ORM layer (Prisma остаётся)
- Prisma переезжает в `packages/db` для шаринга между apps

## Реализация

1. Создать `packages/db` с Prisma schema
2. Создать новый `apps/web` как Next.js 14 App Router проект
3. Мигрировать Prisma schema из `apps/api/prisma` → `packages/db/prisma`
4. Реализовать Server Actions вместо NestJS контроллеров
5. `apps/api` оставить в режиме deprecated (не удалять до полной миграции)
